<h1 id="tuma207-project-management-principles-论述题复习指南">TUMA207 Project Management Principles 论述题复习指南</h1>
<p>基于本文件夹内 30 个 PDF：01-07 主课件、所有 Task/Scenario/Template/Realization、Spotify Model 阅读、Multi-Criteria Analysis 论文。</p>
<p>这门课的考试如果是论述题，重点大概率不是“解释概念是什么”，而是给一个 scenario 后要求你：</p>
<ol>
<li>判断项目处于哪个阶段、主要问题是什么。</li>
<li>识别 stakeholders、目标、限制条件和成功标准。</li>
<li>选择合适的项目管理工具或过程。</li>
<li>说明为什么这样做，而不是机械套模板。</li>
<li>给出可执行的管理动作、沟通方式、责任人和监控指标。</li>
</ol>
<hr>
<h2 id="万能论述题结构">1. 万能论述题结构</h2>
<p>遇到任何 scenario，建议按这个顺序写：</p>
<h3 id="a.-reframe-the-project">A. Reframe the project</h3>
<p>先用 2-3 句话重述项目逻辑：</p>
<ul>
<li>项目目标是什么：business value / public value / operational value。</li>
<li>主要 deliverables 是什么。</li>
<li>关键 constraints 是什么：time, cost, scope, quality, regulation, stakeholder acceptance。</li>
<li>最大 uncertainty 是什么。</li>
</ul>
<p>示例句：</p>
<blockquote>
<p>The project should not be managed only as a technical delivery. It is a value delivery system involving regulatory approval, stakeholder acceptance, operational continuity, and measurable benefits.</p>
</blockquote>
<h3 id="b.-identify-success-criteria">B. Identify success criteria</h3>
<p>不要只说“完成项目”。要把成功标准变成可衡量条件：</p>
<ul>
<li>Scope: deliverables 是否被正式接受。</li>
<li>Schedule: 是否满足 deadline / milestone。</li>
<li>Cost: 是否在 budget baseline 内。</li>
<li>Quality: 是否满足 acceptance criteria / UAT / defect rate。</li>
<li>Stakeholder: sponsor、customer、regulator、community 是否满意。</li>
<li>Benefit: revenue, emission reduction, customer retention, ridership, delivery rate 等是否实现。</li>
</ul>
<h3 id="c.-analyze-stakeholders-and-decision-rights">C. Analyze stakeholders and decision rights</h3>
<p>说明谁有 power、谁有 interest、谁会影响项目成败。</p>
<p>高分点：不要写太泛的 stakeholder。老师在 stakeholder task 提醒过：<code>Government of Singapore</code> 太泛，应该写 <code>Ministry of Transport</code>, <code>LTA</code>, <code>URA</code>, <code>NEA/PUB</code>, <code>Traffic Police</code>, <code>Residents/NGOs</code> 等具体组织。</p>
<h3 id="d.-choose-tools-and-explain-fit">D. Choose tools and explain fit</h3>
<p>根据问题选择工具：</p>
<ul>
<li>目标不清：SMART / OKR / project charter。</li>
<li>角色不清：RACI。</li>
<li>工作范围不清：WBS / scope statement。</li>
<li>时间安排不清：Gantt / schedule baseline / dependencies。</li>
<li>成本不清：cost baseline / EVM。</li>
<li>沟通混乱：communication plan / stakeholder engagement。</li>
<li>风险不清：risk register / probability-impact matrix / ROAM。</li>
<li>供应商不清：SOW / procurement plan / contract control。</li>
<li>项目偏离：status report / RAG / issue log / change request。</li>
<li>项目结束：closeout report / formal acceptance / lessons learned。</li>
</ul>
<h3 id="e.-discuss-trade-offs">E. Discuss trade-offs</h3>
<p>论述题一定要有 trade-off：</p>
<ul>
<li>快 vs 稳：fast tracking 会增加 rework 和 coordination risk。</li>
<li>scope vs time/cost：新需求必须走 change control，否则 scope creep。</li>
<li>stakeholder satisfaction vs schedule：公众参与会增加时间，但降低 resistance。</li>
<li>quality cost vs failure cost：prevention/appraisal 通常比 internal/external failure 便宜。</li>
<li>predictive vs adaptive：requirements stable 适合 predictive；uncertainty 高适合 adaptive；大型基础设施常用 hybrid。</li>
</ul>
<h3 id="f.-define-actions-owners-and-metrics">F. Define actions, owners, and metrics</h3>
<p>最后落到行动：</p>
<ul>
<li>谁负责：RACI / issue owner。</li>
<li>何时完成：milestone / due date。</li>
<li>如何沟通：push, pull, interactive。</li>
<li>如何监控：SPI, CPI, RAG, defect rate, survey, delivery rate, risk review cadence。</li>
</ul>

<h2 id="课程主线">2. 课程主线</h2>
<h3 id="project-performance-principles">01 Project Performance Principles</h3>
<p>可用于论述题的原则：</p>
<ul>
<li>Focus on value: 所有项目活动要回到 business/public value。</li>
<li>Effectively engage stakeholders: stakeholder 会影响 scope, schedule, cost, quality, risk, benefits realization。</li>
<li>Recognize system interactions: 不要孤立看技术问题，要看组织、政策、资源、供应商、用户之间的互动。</li>
<li>Demonstrate leadership: active listening, conflict management, delegation, negotiation, coaching, empathy, decision-making, accountability, adaptability。</li>
<li>Tailor based on context: 不同项目不能套同一模板。</li>
<li>Build quality into process and deliverables: 质量要前置到 planning 和 execution，不是最后检查。</li>
<li>Navigate complexity: human behavior, system behavior, ambiguity, technology innovation。</li>
<li>Optimize risk responses: 理解 risk appetite / threshold。</li>
<li>Embrace adaptability and resilience: 用短反馈循环、prototype、inspection、retrospective 应对变化。</li>
</ul>
<h3 id="project-performance-domains">02 Project Performance Domains</h3>
<p>重点不是背 8 个 domain，而是理解它们并行发生：</p>
<ul>
<li>Stakeholder: identify -&gt; understand/analyze -&gt; prioritize -&gt; engage -&gt; monitor。</li>
<li>Team: culture, trust, open communication, shared ownership, resilience。</li>
<li>Development approach: predictive, adaptive, hybrid。</li>
<li>Planning: scope, schedule, cost, communication, quality, risk 都要形成 baseline。</li>
<li>Project work: 管理工作流、资源、供应商、变更、知识转移。</li>
<li>Delivery: deliverables 要满足 requirements, acceptance criteria, definition of done。</li>
<li>Measurement: 用 baselines 和 metrics 判断项目是否偏离。</li>
<li>Uncertainty: risk, ambiguity, complexity, volatility。</li>
</ul>
<h3 id="initiation">03 Initiation</h3>
<p>Initiation 的论述逻辑：</p>
<ul>
<li>先说明为什么项目值得做：cost-benefit, value, ROI, NPV, IRR, BCR, intangible benefits。</li>
<li>再定义成功：SMART goals / OKR。</li>
<li>再授权项目：project charter。</li>
<li>再识别 stakeholders 和角色：stakeholder analysis / RACI。</li>
<li>避免四类早期失败：unrealistic expectations, miscommunication, lack of resources, scope creep。</li>
</ul>
<h3 id="planning">04 Planning</h3>
<p>Planning 的核心链条：</p>
<p><code>Requirements -&gt; Scope statement -&gt; WBS -&gt; Activities -&gt; Dependencies -&gt; Gantt/Schedule baseline -&gt; Cost baseline -&gt; Communication/Risk/Quality/Procurement plans</code></p>
<p>高分表达：</p>
<blockquote>
<p>A Gantt chart is only meaningful if it is derived from the WBS and if each task has an owner, duration, dependency, and measurable completion criterion.</p>
</blockquote>
<h3 id="execution">05 Execution</h3>
<p>Execution 重点：</p>
<ul>
<li>Tracking: Gantt, roadmap, burndown, status report。</li>
<li>Status report: completed milestones, upcoming tasks, risks/issues, owners, RAG。</li>
<li>Team management: acquire, develop, manage team; conflict management。</li>
<li>Quality: UAT, acceptance criteria, quality tools。</li>
<li>Procurement: bidder conference, proposal evaluation, negotiation, selected sellers, agreements。</li>
<li>Stakeholder management: issue log, change log, management skills。</li>
</ul>
<h3 id="monitoring-and-controlling">06 Monitoring and Controlling</h3>
<p>Monitoring 的逻辑：</p>
<ul>
<li>Compare actual performance against baselines。</li>
<li>Identify variance and root cause。</li>
<li>Decide if corrective action, preventive action, or change request is required。</li>
<li>Communicate status and update documents。</li>
</ul>
<p>核心工具：</p>
<ul>
<li>Validate scope: formal acceptance。</li>
<li>Control scope: prevent scope creep and gold plating。</li>
<li>Control schedule/cost: SPI, CPI, SV, CV。</li>
<li>Control quality: inspection, approved change request review。</li>
<li>Control communication/stakeholder: information needs still met?</li>
<li>Control procurement: seller performance meets agreement?</li>
</ul>
<h3 id="closing">07 Closing</h3>
<p>Closing 不是“结束工作”，而是 formal closure：</p>
<ul>
<li>Confirm deliverables accepted。</li>
<li>Confirm exit criteria met。</li>
<li>Close accounts and contracts。</li>
<li>Handover to operations/support。</li>
<li>Archive records。</li>
<li>Document lessons learned。</li>
<li>Measure stakeholder satisfaction。</li>
<li>Prepare final report with scope, quality, schedule, cost, risks/issues, business needs, future benefits。</li>
</ul>

<h2 id="必背答题框架">4. 必背答题框架</h2>
<h3 id="stakeholder-analysis">4.1 Stakeholder Analysis</h3>
<p>答题步骤：</p>
<ol>
<li>Identify specific stakeholders。</li>
<li>Explain their power: approval, funding, regulation, technical authority, user acceptance, public influence。</li>
<li>Explain their interest: benefits, disruption, compliance, cost, service quality。</li>
<li>Place them in power-interest logic。</li>
<li>Define engagement approach。</li>
</ol>
<p>Power-interest strategy:</p>
<ul>
<li>High power + high interest: manage closely。</li>
<li>High power + low interest: keep satisfied。</li>
<li>Low power + high interest: keep informed。</li>
<li>Low power + low interest: monitor。</li>
</ul>
<p>Singapore EV bus conversion 可写 stakeholders：</p>
<ul>
<li>LTA: owner, fleet electrification target, approvals, operations continuity。</li>
<li>MOT: funding and national transport policy。</li>
<li>Bus operators: daily operation, depot scheduling, driver training, service reliability。</li>
<li>EV conversion contractor / OEM: technical execution, conversion quality, warranty。</li>
<li>NEA: emissions/environmental compliance。</li>
<li>Energy provider / charging infrastructure provider: power capacity, charging uptime。</li>
<li>Commuters: service disruption and reliability。</li>
<li>Drivers/maintenance staff: safety, training, operational adoption。</li>
<li>Residents/businesses near depots: noise, traffic, charging works disruption。</li>
</ul>
<p>高分点：</p>
<blockquote>
<p>The project manager should not treat all stakeholders equally. LTA and MOT require close management because they control approval and funding, while commuters and residents require structured consultation because their support affects operational acceptance and reputational risk.</p>
</blockquote>
<h3 id="smart-goals">4.2 SMART Goals</h3>
<p>Task 中常见 vague goals：</p>
<ul>
<li>Boost brand awareness。</li>
<li>Raise customer retention rate。</li>
</ul>
<p>可改写：</p>
<ul>
<li>Brand awareness: Increase monthly unique visits to the Plant Pals landing page by at least 2,000 by the end of Q4 through the redesigned website, marketing campaign, and print catalogue.</li>
<li>Customer retention: Increase customer retention from 80% to at least 88% by the end of Q4 by launching Plant Pals and implementing an operations/training plan that supports customer satisfaction above 90%.</li>
</ul>
<p>解释时逐项写：</p>
<ul>
<li>Specific: 指明对象、活动、目标群体。</li>
<li>Measurable: 有数字指标。</li>
<li>Attainable: 基于历史 campaign、已有资源或 scenario 信息。</li>
<li>Relevant: 支持 revenue growth / customer retention / business case。</li>
<li>Time-bound: Q3 launch, Q4 assessment。</li>
</ul>
<h3 id="okr">4.3 OKR</h3>
<p>Objective 要 aspirational, aligned, action-oriented, concrete, significant。</p>
<p>Key Results 要 measurable, time-bound, verifiable, results-oriented。</p>
<p>Wonder City 示例：</p>
<p>Objective:</p>
<blockquote>
<p>Provide a reliable and convenient public transportation alternative that connects high-density suburbs to downtown and public resources within two years.</p>
</blockquote>
<p>Key Results:</p>
<ul>
<li>Launch five new bus lines within 24 months while meeting all government regulations。</li>
<li>Ensure the new lines serve at least 50% of the most densely populated areas。</li>
<li>Complete community engagement for all proposed bus line locations before final route approval。</li>
<li>Increase ridership on the served corridors within the first year after launch。</li>
<li>Reduce average waiting time on targeted corridors after implementation。</li>
</ul>
<p>注意：KR 不要写成 task，例如 “hold meetings” 本身不是好 KR，除非它连接到 measurable outcome，如 “obtain documented community feedback from all affected districts before approval”。</p>
<h3 id="raci">4.4 RACI</h3>
<p>规则：</p>
<ul>
<li>Every task needs at least one Responsible。</li>
<li>Every task should have only one Accountable。</li>
<li>Consulted 是提供专业意见或 strong opinion 的人。</li>
<li>Informed 是受影响但不直接做事的人。</li>
</ul>
<p>Plant Pals landing page 示例逻辑：</p>
<ul>
<li>Design landing page: Web Designer R, Web Manager A, Graphic Designer C, others I。</li>
<li>Create mockup: Web Designer R, Web Manager A, Web Developer/Content Writer/Graphic Designer C, QA I。</li>
<li>Design image assets: Graphic Designer R, Web Designer A or Web Manager A depending decision right, others I。</li>
<li>Write content: Content Writer R, Web Manager A, Web Designer C, others I。</li>
<li>Code page: Web Developer R, Web Manager A, QA/Web Designer C, others I。</li>
<li>Test page: QA Tester R, Web Manager A, Web Developer C, others I。</li>
</ul>
<p>Civil Infrastructure 论述重点：</p>
<ul>
<li>LTA 是 overall owner，但不等于每项都 Responsible。</li>
<li>URA 对 land use 有强 authority。</li>
<li>NEA/PUB/Traffic Police 在各自监管领域有 veto power。</li>
<li>Prime Contractor 执行大多数工程任务。</li>
<li>Residents/NGOs 多数是 Consulted，因为他们能影响路线、噪声屏障、公众接受度。</li>
</ul>
<h3 id="wbs---gantt---cost-baseline">4.5 WBS -&gt; Gantt -&gt; Cost Baseline</h3>
<p>答题链条：</p>
<ol>
<li>WBS decomposes scope into manageable work packages。</li>
<li>每个 work package 再变成 activities/tasks。</li>
<li>每个 task 分配 owner、duration、dependency。</li>
<li>Gantt 展示 start/due date 和 sequencing。</li>
<li>Cost baseline 按 schedule 分配预算，方便后续 EVM 监控。</li>
</ol>
<p>Plant Pals Operations &amp; Training WBS:</p>
<ul>
<li>Milestone 1: Establish delivery and logistics plan
<ul>
<li>Source packaging materials</li>
<li>Hire delivery drivers</li>
<li>Calculate delivery fees</li>
</ul>
</li>
<li>Milestone 2: Select and install supply chain software/equipment
<ul>
<li>Supervise vendor software setup</li>
<li>Supervise equipment installation</li>
<li>Determine internal safety protocols</li>
</ul>
</li>
<li>Milestone 3: Develop and launch training program
<ul>
<li>Develop training sessions</li>
<li>Train employees</li>
<li>Monitor progress and improve training</li>
</ul>
</li>
</ul>
<p>高分点：</p>
<blockquote>
<p>A WBS is not just a list of tasks. It is a scope control tool. If work is not in the WBS/scope baseline, it should not enter the schedule or budget without change control.</p>
</blockquote>
<h3 id="communication-plan">4.6 Communication Plan</h3>
<p>Planning questions from slides:</p>
<ul>
<li>Who needs information?</li>
<li>What information does each stakeholder need?</li>
<li>Why should information be shared?</li>
<li>What is the best way to provide information?</li>
<li>When and how often is information needed?</li>
<li>Who has the information needed?</li>
</ul>
<p>Communication methods:</p>
<ul>
<li>Interactive: meetings, calls, workshops. 适合 high power/high interest 或需要协商的问题。</li>
<li>Push: emails, reports, memos. 适合 status update、instructions。</li>
<li>Pull: repository, dashboard, intranet. 适合大量信息、模板、记录。</li>
</ul>
<p>Plant Pals training communication logic：</p>
<ul>
<li>Core team: daily in-person planning meetings。</li>
<li>Vendor: one-time half-day interactive meeting before SOW ends。</li>
<li>HR: weekly/as-needed check-ins。</li>
<li>Facilities/Print Shop: weekly planning check-ins。</li>
<li>Employees: weekly official email plus day-before reminders。</li>
<li>Manager: weekly high-level status update。</li>
<li>Senior leadership: one-time email outcome report after training。</li>
</ul>
<p>高分点：</p>
<blockquote>
<p>Communication frequency should reflect stakeholder involvement and decision needs. Senior leaders need concise exception-based updates, while the core delivery team needs frequent interactive coordination.</p>
</blockquote>
<h3 id="risk-management">4.7 Risk Management</h3>
<p>Risk answer structure：</p>
<ol>
<li>Identify risk event, not vague category。</li>
<li>Estimate probability。</li>
<li>Estimate impact。</li>
<li>Assign inherent risk using probability-impact matrix。</li>
<li>Choose response。</li>
<li>Assign owner and monitoring trigger。</li>
</ol>
<p>Negative risk responses：</p>
<ul>
<li>Avoid: eliminate the threat。</li>
<li>Transfer: shift impact/ownership, often contract/insurance。</li>
<li>Mitigate/reduce/control: reduce probability or impact。</li>
<li>Accept: no proactive action, monitor or contingency。</li>
<li>Escalate: outside PM authority。</li>
</ul>
<p>Plant Pals risk examples：</p>
<ul>
<li>Delivery trucks cost $32,000 more, 50% probability: high probability + high impact -&gt; high risk; reduce/control through revised quotes, alternative suppliers, budget reserve, sponsor approval。</li>
<li>Vendor higher rate, 20-25% probability, $17,000 impact: medium risk; negotiate fixed price or transfer through contract terms。</li>
<li>Lost shipment, 5% probability, $25,000 impact: low probability but high impact; transfer through insurance/vendor liability, mitigation through tracking and backup stock。</li>
<li>Not enough employees before training, 20-30% probability, &gt;1 month delay: medium/high schedule risk; mitigate by early recruitment, backup trainers, phased training。</li>
</ul>
<h3 id="sow--procurement">4.8 SOW / Procurement</h3>
<p>SOW 必须写清楚：</p>
<ul>
<li>Purpose。</li>
<li>In-scope work。</li>
<li>Out-of-scope work。</li>
<li>Deliverables。</li>
<li>Milestones。</li>
<li>Estimated hours / completion date。</li>
<li>Payment terms。</li>
</ul>
<p>Plant Pals vendor SOW:</p>
<p>In scope:</p>
<ul>
<li>Configure inventory and fulfillment tracking software。</li>
<li>Audit existing software and optimize。</li>
<li>Install software on devices/equipment。</li>
<li>Install warehouse fulfillment equipment。</li>
<li>Create training manuals and maintenance guide。</li>
<li>Meet Training Manager and HR Specialist。</li>
</ul>
<p>Out of scope:</p>
<ul>
<li>Training all employees。</li>
<li>Ongoing maintenance。</li>
<li>Separate future software maintenance contract。</li>
</ul>
<p>高分点：</p>
<blockquote>
<p>Out-of-scope items are as important as in-scope items because they prevent scope creep, clarify payment expectations, and reduce procurement disputes.</p>
</blockquote>
<h3 id="status-report--rag--roam">4.9 Status Report / RAG / ROAM</h3>
<p>Status report components：</p>
<ul>
<li>Completed milestones: date, owner, comment。</li>
<li>Upcoming milestones: due date, owner, dependency。</li>
<li>Top risks/issues: impact, action, owner。</li>
<li>Overall RAG。</li>
</ul>
<p>Plant Pals test batch issues：</p>
<ul>
<li>10% plants not properly potted -&gt; complaints, profit loss, budget issue -&gt; Warehouse Operations Manager owns action -&gt; evaluate and adjust potting process。</li>
<li>Software receives only 30% of requests/complaints -&gt; customer dissatisfaction -&gt; IT Specialist owns action -&gt; fix software/routing and manual workaround。</li>
<li>Delivery completion only 80% vs target 95% -&gt; cancellations -&gt; HR Specialist owns action -&gt; hire/assign drivers, temporary delivery support。</li>
</ul>
<p>RAG:</p>
<ul>
<li>The task explicitly expects Amber: project is off-track but correctable with targeted actions。</li>
</ul>
<p>ROAM:</p>
<ul>
<li>Resolved: issue eliminated。</li>
<li>Owned: assigned to someone and monitored。</li>
<li>Accepted: minor or unavoidable, team works around it。</li>
<li>Mitigated: action taken to reduce impact/probability。</li>
</ul>
<p>高分点：</p>
<blockquote>
<p>Reporting an issue is not enough. A good project status report connects issue -&gt; impact -&gt; owner -&gt; action -&gt; deadline -&gt; communication.</p>
</blockquote>
<h3 id="evm--spi--cpi">4.10 EVM / SPI / CPI</h3>
<p>Formulas:</p>
<ul>
<li>PV = BAC x planned % complete。</li>
<li>EV = BAC x actual % complete。</li>
<li>AC = actual cost。</li>
<li>CV = EV - AC。</li>
<li>SV = EV - PV。</li>
<li>CPI = EV / AC。</li>
<li>SPI = EV / PV。</li>
</ul>
<p>Interpretation:</p>
<ul>
<li>SPI = 1: on schedule; SPI &lt; 1: behind; SPI &gt; 1: ahead。</li>
<li>CPI = 1: on budget; CPI &lt; 1: over budget; CPI &gt; 1: under budget。</li>
<li>CV &lt; 0: over budget。</li>
<li>SV &lt; 0: behind schedule。</li>
</ul>
<p>Template example:</p>
<ul>
<li>4 phases, $10,000 each, total BAC = $40,000。</li>
<li>End of month 3, planned phases 1-3 complete, PV = $30,000。</li>
<li>Actual: requirement complete $10,000; architecture complete $12,000; development 50% complete $9,000; system testing not started。</li>
<li>EV = 10,000 + 10,000 + 5,000 = $25,000。</li>
<li>AC = 10,000 + 12,000 + 9,000 = $31,000。</li>
<li>CV = 25,000 - 31,000 = -$6,000。</li>
<li>SV = 25,000 - 30,000 = -$5,000。</li>
<li>CPI = 25,000 / 31,000 = 0.81。</li>
<li>SPI = 25,000 / 30,000 = 0.83。</li>
</ul>
<p>Conclusion:</p>
<blockquote>
<p>The project is both over budget and behind schedule. The project manager should analyze the causes in architecture and development, update forecasts, communicate Amber/Red status, and propose corrective actions or change requests rather than silently absorbing the variance.</p>
</blockquote>
<h3 id="closing--closeout-report">4.11 Closing / Closeout Report</h3>
<p>Closeout report 应包含：</p>
<ul>
<li>Executive summary。</li>
<li>Key accomplishments。</li>
<li>Lessons learned: what went well, what went wrong, how resolved。</li>
<li>Open items。</li>
<li>Next steps and future considerations。</li>
<li>Project timeline。</li>
<li>Resources and archive。</li>
</ul>
<p>Plant Pals 可写 lessons learned：</p>
<p>Went well:</p>
<ul>
<li>Logistics plan reduced delivery cost by 50 cents per unit。</li>
<li>Software/equipment improved procurement and ordering efficiency。</li>
<li>Training over 75% employees before launch improved readiness。</li>
<li>Customer survey feedback improved on-time delivery by 10% and customer service satisfaction by more than 42%。</li>
</ul>
<p>Went wrong and resolution:</p>
<ul>
<li>Software installation delayed by three days -&gt; improve vendor testing and installation buffer。</li>
<li>Plant potting defects caused complaints -&gt; add quality checklist and warehouse QA before shipment。</li>
<li>Driver shortage reduced delivery rate -&gt; earlier HR planning, backup drivers, phased delivery plan。</li>
</ul>
<p>Open items:</p>
<ul>
<li>Improve on-time delivery to 95%。</li>
<li>Continue customer satisfaction survey。</li>
<li>Contract ongoing software maintenance separately。</li>
</ul>
<p>高分点：</p>
<blockquote>
<p>Closing should transfer knowledge and operational ownership, not only record completion.</p>
</blockquote>
<h3 id="spotify-model--tailoring">4.12 Spotify Model / Tailoring</h3>
<p>Spotify model:</p>
<ul>
<li>Squad: autonomous product team, similar to Scrum team。</li>
<li>Tribe: collection of squads in related area。</li>
<li>Chapter: people with similar skills within a tribe。</li>
<li>Guild: broader community of practice across organization。</li>
</ul>
<p>考试逻辑：</p>
<ul>
<li>不要说“Spotify 模型最好”。</li>
<li>应说：Spotify demonstrates adaptability, autonomy, and scaling, but copying it blindly is wrong。</li>
<li>Tailor based on team size, culture, product modularity, communication needs, and organizational capability。</li>
</ul>
<p>高分句：</p>
<blockquote>
<p>The lesson from Spotify is not the labels of squads and tribes, but the principle of balancing autonomy with alignment as the organization scales.</p>
</blockquote>
<h3 id="multi-criteria-analysis-mca">4.13 Multi-Criteria Analysis (MCA)</h3>
<p>MCA 适合论述“如何在多个冲突标准中选择方案”。</p>
<p>步骤：</p>
<ol>
<li>Define alternatives。</li>
<li>Define criteria。</li>
<li>Collect reliable data。</li>
<li>Score each alternative under each criterion。</li>
<li>Weight criteria based on importance。</li>
<li>Calculate weighted score。</li>
<li>Rank alternatives。</li>
<li>Discuss sensitivity and fairness。</li>
</ol>
<p>MCA paper 中 criteria：</p>
<ul>
<li>Skid resistance: safety。</li>
<li>Surface water drainage: safety。</li>
<li>Noise: comfort。</li>
<li>Pavement roughness: safety and comfort。</li>
</ul>
<p>高分点：</p>
<blockquote>
<p>MCA does not remove judgment; it makes assumptions visible. The fairness of the result depends on transparent scoring, justified weighting, and comparable measurement units.</p>
</blockquote>

<h2 id="高频论述题预测与答题要点">5. 高频论述题预测与答题要点</h2>
<h3 id="q1.-a-public-authority-plans-to-convert-ice-buses-to-evs-without-disrupting-operations.-identify-key-stakeholders-and-propose-an-engagement-strategy.">Q1. A public authority plans to convert ICE buses to EVs without disrupting operations. Identify key stakeholders and propose an engagement strategy.</h3>
<p>答题要点：</p>
<ul>
<li>Value: emission reduction, cleaner fleet, operational continuity。</li>
<li>Stakeholders: LTA, MOT, bus operators, commuters, contractors/OEMs, charging provider, NEA, depot managers, drivers, maintenance staff, residents。</li>
<li>Power-interest: LTA/MOT high power; bus operators high interest; commuters/residents high interest but lower formal power; regulators high domain veto power。</li>
<li>Engagement: close management for LTA/MOT/operators; consultation with commuters/residents; technical workshops with contractors; training communication for drivers。</li>
<li>Risk: service disruption, charging capacity, safety certification, budget, public dissatisfaction。</li>
<li>Metrics: number of buses converted, service reliability, emissions reduction, downtime, safety incidents。</li>
</ul>
<h3 id="q2.-the-sponsor-gives-vague-goals-such-as-“increase-awareness”-and-“improve-retention”.-convert-them-into-smart-goals-and-justify.">Q2. The sponsor gives vague goals such as “increase awareness” and “improve retention”. Convert them into SMART goals and justify.</h3>
<p>答题要点：</p>
<ul>
<li>Include metric, target, deadline, relevance。</li>
<li>Explain each SMART component。</li>
<li>Link to business objective, not only marketing activity。</li>
</ul>
<h3 id="q3.-wonder-city-wants-five-new-bus-lines-in-two-years.-create-okrs.">Q3. Wonder City wants five new bus lines in two years. Create OKRs.</h3>
<p>答题要点：</p>
<ul>
<li>Objective should be aspirational but concrete。</li>
<li>KR should include 5 lines, 24 months, regulatory compliance, 50% dense area coverage, community buy-in, wait time/ridership。</li>
<li>Avoid writing only tasks。</li>
</ul>
<h3 id="q4.-a-project-team-has-many-roles-and-tasks.-how-would-you-use-raci-to-prevent-confusion">Q4. A project team has many roles and tasks. How would you use RACI to prevent confusion?</h3>
<p>答题要点：</p>
<ul>
<li>Explain R/A/C/I logic。</li>
<li>Only one accountable per task。</li>
<li>RACI prevents duplicated work, missing decision rights, delayed approvals。</li>
<li>Apply to scenario with examples。</li>
<li>Discuss limitation: RACI does not replace communication plan。</li>
</ul>
<h3 id="q5.-explain-how-a-wbs-becomes-a-gantt-chart-and-then-a-cost-baseline.">Q5. Explain how a WBS becomes a Gantt chart and then a cost baseline.</h3>
<p>答题要点：</p>
<ul>
<li>Scope decomposition。</li>
<li>Activities/tasks and milestones。</li>
<li>Dependencies, owners, durations。</li>
<li>Schedule baseline。</li>
<li>Cost aggregation by task/week。</li>
<li>Baseline later supports monitoring and EVM。</li>
</ul>
<h3 id="q6.-in-plant-pals-training-different-stakeholders-need-different-communications.-design-a-communication-plan.">Q6. In Plant Pals training, different stakeholders need different communications. Design a communication plan.</h3>
<p>答题要点：</p>
<ul>
<li>Core team daily interactive。</li>
<li>Vendor one-time detailed meeting before SOW end。</li>
<li>HR weekly/as-needed。</li>
<li>Facilities/Print Shop weekly planning check-ins。</li>
<li>Employees push updates/reminders and training sessions。</li>
<li>Manager weekly summary。</li>
<li>Senior leadership post-training outcome email。</li>
<li>Justify based on involvement and information need。</li>
</ul>
<h3 id="q7.-given-budget-and-schedule-risks-prioritize-and-select-responses.">Q7. Given budget and schedule risks, prioritize and select responses.</h3>
<p>答题要点：</p>
<ul>
<li>Probability-impact matrix。</li>
<li>High probability + high impact gets immediate mitigation。</li>
<li>Avoid/transfer/reduce/accept must match scenario。</li>
<li>Assign risk owner and trigger。</li>
<li>Include contingency reserve if appropriate。</li>
</ul>
<h3 id="q8.-the-project-is-off-track-software-receives-only-30-complaints-delivery-rate-is-80-10-plants-poorly-potted.-write-status-logic.">Q8. The project is off track: software receives only 30% complaints, delivery rate is 80%, 10% plants poorly potted. Write status logic.</h3>
<p>答题要点：</p>
<ul>
<li>RAG Amber。</li>
<li>Separate issues from risks。</li>
<li>Explain impact: customer dissatisfaction, cancellations, budget loss。</li>
<li>Owners: IT Specialist, HR Specialist, Warehouse Operations Manager。</li>
<li>Actions: software fix/workaround, hire drivers, adjust potting QA。</li>
<li>Communication: sponsor gets concise status; team gets action log。</li>
</ul>
<h3 id="q9.-a-vendor-will-install-software-and-equipment.-what-belongs-in-the-sow">Q9. A vendor will install software and equipment. What belongs in the SOW?</h3>
<p>答题要点：</p>
<ul>
<li>Purpose, scope, out-of-scope, deliverables, milestones, hours, due date, payment。</li>
<li>Explicitly exclude employee training and maintenance if not contracted。</li>
<li>Link SOW to procurement control and dispute prevention。</li>
</ul>
<h3 id="q10.-evm-shows-cpi--1-and-spi--1.-what-should-the-project-manager-do">Q10. EVM shows CPI &lt; 1 and SPI &lt; 1. What should the project manager do?</h3>
<p>答题要点：</p>
<ul>
<li>Interpret: over budget and behind schedule。</li>
<li>Diagnose root cause。</li>
<li>Update forecast: EAC/ETC/VAC if data available。</li>
<li>Decide corrective action: re-sequence, add resources, reduce scope through change control, vendor escalation。</li>
<li>Communicate via status report。</li>
<li>Avoid hiding variance。</li>
</ul>
<h3 id="q11.-a-project-has-stakeholder-pressure-to-add-features-after-scope-baseline.-how-should-pm-respond">Q11. A project has stakeholder pressure to add features after scope baseline. How should PM respond?</h3>
<p>答题要点：</p>
<ul>
<li>Recognize scope creep/gold plating。</li>
<li>Compare request against business value and acceptance criteria。</li>
<li>Use change request and integrated change control。</li>
<li>Assess schedule/cost/risk/quality impact。</li>
<li>Communicate decision and update baselines if approved。</li>
</ul>
<h3 id="q12.-the-project-is-technically-successful-but-users-are-dissatisfied.-was-it-successful">Q12. The project is technically successful but users are dissatisfied. Was it successful?</h3>
<p>答题要点：</p>
<ul>
<li>Not necessarily。</li>
<li>Project success includes outcomes, benefits, stakeholder satisfaction, adoption, quality。</li>
<li>Need requirements traceability, UAT, stakeholder engagement, acceptance criteria。</li>
<li>Closing report should document gap and future benefits plan。</li>
</ul>
<h3 id="q13.-should-a-civil-infrastructure-project-use-predictive-adaptive-or-hybrid-approach">Q13. Should a civil infrastructure project use predictive, adaptive, or hybrid approach?</h3>
<p>答题要点：</p>
<ul>
<li>Predictive for regulatory approvals, construction sequence, safety requirements, contract milestones。</li>
<li>Adaptive for smart technology, community feedback, digital systems, route refinements。</li>
<li>Hybrid is strongest answer。</li>
<li>Explain tailoring based on requirements certainty, risk, regulation, stakeholder feedback。</li>
</ul>
<h3 id="q14.-how-can-mca-support-project-decision-making">Q14. How can MCA support project decision-making?</h3>
<p>答题要点：</p>
<ul>
<li>Use when criteria conflict。</li>
<li>Define alternatives and criteria。</li>
<li>Score, weight, rank。</li>
<li>Explain assumptions and sensitivity。</li>
<li>Example: pavement method, vendor selection, route selection, technology option。</li>
<li>Do not claim MCA is purely objective; weighting is judgment-based。</li>
</ul>
<h3 id="q15.-what-should-be-included-in-a-project-closeout-report-and-why">Q15. What should be included in a project closeout report and why?</h3>
<p>答题要点：</p>
<ul>
<li>Formal acceptance and transition。</li>
<li>Summary of scope, quality, schedule, cost, risks/issues。</li>
<li>Key accomplishments。</li>
<li>Lessons learned。</li>
<li>Open items。</li>
<li>Next steps。</li>
<li>Archive/resources。</li>
<li>Value: future PMs can reuse knowledge and avoid repeating mistakes。</li>
</ul>

<h2 id="常见失分点">7. 常见失分点</h2>
<ul>
<li>只列定义，不说明如何用于 scenario。</li>
<li>Stakeholder 写得太泛，如 “government”, “engineering team”。</li>
<li>SMART goal 没有数字或 deadline。</li>
<li>OKR 的 key result 写成 task，而不是 measurable result。</li>
<li>RACI 每个 task 有多个 Accountable。</li>
<li>WBS、Gantt、Cost baseline 互相脱节。</li>
<li>Communication plan 对所有人同样频率。</li>
<li>高风险只写 accept，没有解释 risk appetite 或 contingency。</li>
<li>Status report 只有问题，没有 owner/action/deadline。</li>
<li>EVM 只计算 CPI/SPI，不解释管理动作。</li>
<li>Closing 只写“项目完成”，没有 formal acceptance、handover、archive、lessons learned。</li>
<li>看到 Spotify 就照搬 squads/tribes，没有讨论 tailoring。</li>
<li>MCA 只给排名，没有解释 scoring/weighting 的主观性和可比性。</li>
</ul>
