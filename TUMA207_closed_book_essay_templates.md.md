<h1 id="tuma207-闭卷论述题万能句与答题模板">TUMA207 闭卷论述题万能句与答题模板</h1>
<p>这份材料的定位：不要把课堂 Scenario 当作题库。Scenario 只说明老师喜欢考什么能力：在一个项目情境里判断问题、选择工具、解释理由、提出行动。闭卷考试更需要背的是“通用逻辑”和“可迁移句型”。</p>
<p>建议背诵目标：</p>
<ol>
<li>会用固定结构组织任何论述题。</li>
<li>会把概念转成管理动作，而不是只解释定义。</li>
<li>会写出 trade-off、stakeholder、baseline、risk、communication、monitoring。</li>
<li>即使忘记具体细节，也能写出有项目管理逻辑的答案。</li>
</ol>
<hr>
<h2 id="闭卷论述题总原则">1. 闭卷论述题总原则</h2>
<p>任何题目都尽量回答这五件事：</p>
<ol>
<li>
<p>What is the project trying to achieve?<br>
项目要创造什么 value。</p>
</li>
<li>
<p>Who can affect or be affected by the project?<br>
谁是 stakeholder，谁有 decision power。</p>
</li>
<li>
<p>What needs to be planned or controlled?<br>
scope, schedule, cost, quality, risk, communication。</p>
</li>
<li>
<p>What tool/process should be used and why?<br>
不只点名工具，要解释 fit。</p>
</li>
<li>
<p>What action should the project manager take?<br>
owner, timeline, metric, communication, monitoring。</p>
</li>
</ol>
<p>核心记忆句：</p>
<blockquote>
<p>A good project management answer should connect value, stakeholders, baselines, risks, communication, and corrective action.</p>
</blockquote>
<hr>
<h2 id="万能六段式答题模板">2. 万能六段式答题模板</h2>
<p>适用于几乎所有长论述题。</p>
<h3 id="paragraph-1-project-framing">Paragraph 1: Project Framing</h3>
<p>作用：先告诉老师你看懂了题目的项目本质。</p>
<p>万能句：</p>
<blockquote>
<p>I would first frame this project as a value delivery problem, not only as a technical or administrative task.</p>
</blockquote>
<blockquote>
<p>The main purpose of the project is to deliver measurable value while balancing scope, schedule, cost, quality, risk, and stakeholder expectations.</p>
</blockquote>
<p>中文思路：</p>
<ul>
<li>这个项目不是单纯完成任务，而是要交付 value。</li>
<li>value 可能是 revenue、customer satisfaction、emission reduction、public service、operational efficiency。</li>
<li>先说清楚项目目标和限制条件。</li>
</ul>
<p>可套模板：</p>
<blockquote>
<p>In this case, the project aims to achieve [main value]. Therefore, the project manager should define success not only by completing deliverables, but also by whether the project meets [business/public/operational] objectives.</p>
</blockquote>
<h3 id="paragraph-2-success-criteria-and-objectives">Paragraph 2: Success Criteria and Objectives</h3>
<p>作用：把 vague goal 变成可评估的目标。</p>
<p>万能句：</p>
<blockquote>
<p>The first management step is to translate the broad goal into clear success criteria.</p>
</blockquote>
<blockquote>
<p>Success should be measured through both delivery metrics and outcome metrics.</p>
</blockquote>
<blockquote>
<p>A deliverable is not successful simply because it is finished; it must be accepted and must support the intended benefits.</p>
</blockquote>
<p>可写内容：</p>
<ul>
<li>Scope: deliverables and acceptance criteria。</li>
<li>Schedule: milestone or deadline。</li>
<li>Cost: budget or cost baseline。</li>
<li>Quality: defect rate, UAT, compliance, fitness for use。</li>
<li>Stakeholder: satisfaction, buy-in, formal approval。</li>
<li>Benefits: revenue, retention, ridership, emission reduction, efficiency。</li>
</ul>
<p>可套模板：</p>
<blockquote>
<p>I would define success using SMART criteria: [specific target], measured by [metric], achievable with [resources/constraints], relevant to [business value], and assessed by [deadline].</p>
</blockquote>
<h3 id="paragraph-3-stakeholders-and-roles">Paragraph 3: Stakeholders and Roles</h3>
<p>作用：显示你知道项目由人推动，stakeholder 会影响结果。</p>
<p>万能句：</p>
<blockquote>
<p>Stakeholder management is critical because stakeholders can influence requirements, funding, approvals, risks, acceptance, and benefits realization.</p>
</blockquote>
<blockquote>
<p>The project manager should not treat all stakeholders equally; engagement should be prioritized by power, interest, influence, and impact.</p>
</blockquote>
<blockquote>
<p>Clear decision rights are necessary to avoid delays, duplicated work, and unclear accountability.</p>
</blockquote>
<p>可写内容：</p>
<ul>
<li>Who has approval power?</li>
<li>Who provides funding/resources?</li>
<li>Who performs the work?</li>
<li>Who is affected by the result?</li>
<li>Who can block or accelerate the project?</li>
<li>Who should be Responsible, Accountable, Consulted, Informed?</li>
</ul>
<p>可套模板：</p>
<blockquote>
<p>I would use a power-interest analysis to identify which stakeholders should be managed closely, kept satisfied, kept informed, or monitored. For execution, I would use a RACI chart so that each major task has at least one Responsible person and only one Accountable person.</p>
</blockquote>
<h3 id="paragraph-4-planning-and-tools">Paragraph 4: Planning and Tools</h3>
<p>作用：把项目管理工具串成逻辑链。</p>
<p>万能句：</p>
<blockquote>
<p>The project plan should be derived from the scope, not from an arbitrary list of activities.</p>
</blockquote>
<blockquote>
<p>The WBS converts project scope into manageable work packages; the Gantt chart sequences those work packages over time; the cost baseline allocates the budget across the schedule.</p>
</blockquote>
<blockquote>
<p>Planning is important because it creates baselines against which actual performance can later be measured.</p>
</blockquote>
<p>可写内容：</p>
<ul>
<li>Project charter: authorization and high-level objectives。</li>
<li>WBS: scope decomposition。</li>
<li>Gantt: sequence, owner, duration, dependency。</li>
<li>Cost baseline: planned spending over time。</li>
<li>Communication plan: who needs what information。</li>
<li>Risk register: probability, impact, response。</li>
<li>SOW: vendor scope, deliverables, exclusions。</li>
</ul>
<p>可套模板：</p>
<blockquote>
<p>I would start with the project charter and scope statement, decompose the work into a WBS, assign owners and dependencies, develop a Gantt chart, and then establish schedule and cost baselines for monitoring.</p>
</blockquote>
<h3 id="paragraph-5-risk-communication-and-control">Paragraph 5: Risk, Communication, and Control</h3>
<p>作用：显示你不会只做计划，还会应对变化。</p>
<p>万能句：</p>
<blockquote>
<p>Because every project contains uncertainty, the project manager should identify risks early and review them throughout the project.</p>
</blockquote>
<blockquote>
<p>Risks should be assessed by probability and impact, then assigned an appropriate response such as avoid, transfer, mitigate, accept, or escalate.</p>
</blockquote>
<blockquote>
<p>Communication should be tailored to stakeholder needs; senior leaders usually need concise status and exceptions, while the project team needs frequent coordination.</p>
</blockquote>
<blockquote>
<p>Once execution begins, actual performance should be compared against the baselines.</p>
</blockquote>
<p>可写内容：</p>
<ul>
<li>Risk register / probability-impact matrix。</li>
<li>Issue log。</li>
<li>Change request。</li>
<li>RAG status。</li>
<li>SPI/CPI。</li>
<li>Status report。</li>
<li>Stakeholder communication。</li>
</ul>
<p>可套模板：</p>
<blockquote>
<p>If performance deviates from the baseline, I would first diagnose the root cause, then decide whether corrective action, preventive action, or a formal change request is required.</p>
</blockquote>
<h3 id="paragraph-6-conclusion--pm-action">Paragraph 6: Conclusion / PM Action</h3>
<p>作用：最后收束到项目经理的管理动作。</p>
<p>万能句：</p>
<blockquote>
<p>Therefore, the project manager’s role is to integrate people, process, data, and decisions so that the project remains aligned with its intended value.</p>
</blockquote>
<blockquote>
<p>The best response is not simply to complete tasks, but to maintain alignment among objectives, stakeholders, baselines, risks, and benefits.</p>
</blockquote>
<blockquote>
<p>A strong project manager should make assumptions visible, communicate trade-offs clearly, and use evidence to support decisions.</p>
</blockquote>
<p>可套模板：</p>
<blockquote>
<p>In conclusion, I would manage this project by clarifying objectives, engaging the right stakeholders, establishing realistic baselines, monitoring performance, and taking corrective action when the project no longer supports its intended value.</p>
</blockquote>
<hr>
<h2 id="考场-30-秒审题法">3. 考场 30 秒审题法</h2>
<p>看到题目后，不要马上写。先在草稿上写 6 个词：</p>
<p><code>Value -&gt; Stakeholders -&gt; Scope -&gt; Baseline -&gt; Risk -&gt; Action</code></p>
<p>然后快速回答：</p>
<ul>
<li>Value: 这个项目为什么要做？</li>
<li>Stakeholders: 谁能影响成功？</li>
<li>Scope: 要交付什么？不交付什么？</li>
<li>Baseline: 用什么衡量计划和实际差异？</li>
<li>Risk: 最大不确定性是什么？</li>
<li>Action: PM 下一步做什么？</li>
</ul>
<p>如果题目问得很宽，就用这 6 个词搭答案。</p>

<h2 id="论述题万能开头">4. 论述题万能开头</h2>
<h3 id="开头-1：价值导向型">开头 1：价值导向型</h3>
<blockquote>
<p>I would approach this project from a value delivery perspective. The key question is not only whether the deliverables are completed, but whether they create the intended business, operational, or public value.</p>
</blockquote>
<p>适合：</p>
<ul>
<li>问项目成功。</li>
<li>问为什么需要项目管理。</li>
<li>问如何管理一个复杂项目。</li>
</ul>
<h3 id="开头-2：阶段判断型">开头 2：阶段判断型</h3>
<blockquote>
<p>The appropriate project management response depends on the project phase. In initiation, the focus is on objectives and authorization; in planning, it is on baselines; in execution, it is on coordination; in monitoring and controlling, it is on variance and change; and in closing, it is on acceptance and knowledge transfer.</p>
</blockquote>
<p>适合：</p>
<ul>
<li>问 project life cycle。</li>
<li>问如何从头到尾管理。</li>
<li>问某个过程为什么重要。</li>
</ul>
<h3 id="开头-3：问题诊断型">开头 3：问题诊断型</h3>
<blockquote>
<p>The main issue in this situation is not only the visible symptom, but the lack of alignment among objectives, responsibilities, communication, and control mechanisms.</p>
</blockquote>
<p>适合：</p>
<ul>
<li>项目延误。</li>
<li>成本超支。</li>
<li>stakeholder conflict。</li>
<li>质量问题。</li>
</ul>
<h3 id="开头-4：工具选择型">开头 4：工具选择型</h3>
<blockquote>
<p>I would not use a project management tool as a template only. I would select the tool based on the problem it solves and the decision it supports.</p>
</blockquote>
<p>适合：</p>
<ul>
<li>问 WBS、RACI、Gantt、Risk Register、SOW、Status Report。</li>
</ul>

<h2 id="高频主题万能句库">5. 高频主题万能句库</h2>
<h3 id="project-value">5.1 Project Value</h3>
<blockquote>
<p>Project success should be evaluated by both outputs and outcomes.</p>
</blockquote>
<blockquote>
<p>Outputs are the deliverables produced by the project, while outcomes are the benefits and value enabled by those deliverables.</p>
</blockquote>
<blockquote>
<p>A project can be delivered on time and still fail if it does not meet stakeholder needs or business objectives.</p>
</blockquote>
<blockquote>
<p>Value may be financial, operational, environmental, social, or strategic depending on the project context.</p>
</blockquote>
<h3 id="initiation">5.2 Initiation</h3>
<blockquote>
<p>Initiation is important because it clarifies why the project should exist before detailed planning begins.</p>
</blockquote>
<blockquote>
<p>A project charter gives formal authorization and aligns the project manager, sponsor, and stakeholders on high-level objectives.</p>
</blockquote>
<blockquote>
<p>Without clear initiation, the project may suffer from unrealistic expectations, unclear scope, resource shortages, or miscommunication.</p>
</blockquote>
<blockquote>
<p>Cost-benefit analysis helps determine whether the expected benefits justify the required investment and risk.</p>
</blockquote>
<h3 id="smart-goals">5.3 SMART Goals</h3>
<blockquote>
<p>A goal is useful only when the project team can understand it, measure it, and know when it has been achieved.</p>
</blockquote>
<blockquote>
<p>SMART goals reduce ambiguity by defining the target, metric, feasibility, relevance, and deadline.</p>
</blockquote>
<blockquote>
<p>The goal should be measurable enough to guide decisions, not just inspirational.</p>
</blockquote>
<h3 id="okr">5.4 OKR</h3>
<blockquote>
<p>Objectives describe what the project is trying to achieve; key results define how success will be measured.</p>
</blockquote>
<blockquote>
<p>A good objective should be aspirational and aligned with organizational goals, while key results should be measurable and time-bound.</p>
</blockquote>
<blockquote>
<p>Key results should describe outcomes, not just activities.</p>
</blockquote>
<h3 id="stakeholders">5.5 Stakeholders</h3>
<blockquote>
<p>Projects are performed by people and for people, so stakeholder engagement is a core part of project success.</p>
</blockquote>
<blockquote>
<p>Stakeholders may affect scope, schedule, cost, risk, quality, acceptance, and benefits realization.</p>
</blockquote>
<blockquote>
<p>The influence and interest of stakeholders may change during the project, so stakeholder analysis should be reviewed throughout the life cycle.</p>
</blockquote>
<blockquote>
<p>High-power stakeholders require close management because they can approve, block, fund, or redirect the project.</p>
</blockquote>
<h3 id="raci">5.6 RACI</h3>
<blockquote>
<p>RACI reduces confusion by separating who does the work from who owns the final decision.</p>
</blockquote>
<blockquote>
<p>Each task should have at least one Responsible person, but only one Accountable person.</p>
</blockquote>
<blockquote>
<p>Too many Accountable roles create confusion; too many Consulted roles slow decision-making.</p>
</blockquote>
<blockquote>
<p>RACI is especially useful when several departments or external parties contribute to the same deliverable.</p>
</blockquote>
<h3 id="wbs">5.7 WBS</h3>
<blockquote>
<p>A WBS breaks the project scope into smaller and more manageable work packages.</p>
</blockquote>
<blockquote>
<p>The 100% rule means that the WBS should include all work required to deliver the project scope, but not unrelated work.</p>
</blockquote>
<blockquote>
<p>WBS is a scope management tool, not just a task list.</p>
</blockquote>
<blockquote>
<p>If work is not included in the approved scope baseline, it should not be added without change control.</p>
</blockquote>
<h3 id="schedule--gantt">5.8 Schedule / Gantt</h3>
<blockquote>
<p>A Gantt chart helps the project team understand task sequence, timing, ownership, and deadlines.</p>
</blockquote>
<blockquote>
<p>A schedule is only reliable if task dependencies, resource availability, and realistic durations are considered.</p>
</blockquote>
<blockquote>
<p>Fast tracking may shorten the schedule but can increase coordination risk and rework.</p>
</blockquote>
<blockquote>
<p>Schedule compression should be evaluated against cost, risk, and quality impacts.</p>
</blockquote>
<h3 id="cost">5.9 Cost</h3>
<blockquote>
<p>Cost planning converts estimates into a cost baseline that can be monitored during execution.</p>
</blockquote>
<blockquote>
<p>A budget should include contingency reserves for known-unknown risks.</p>
</blockquote>
<blockquote>
<p>Cost variance should be analyzed together with schedule and scope, because delays and rework often create cost overruns.</p>
</blockquote>
<h3 id="quality">5.10 Quality</h3>
<blockquote>
<p>Quality should be planned and built into the process, not inspected only at the end.</p>
</blockquote>
<blockquote>
<p>Quality means fitness for use and conformance to requirements; grade refers to the level of features or class.</p>
</blockquote>
<blockquote>
<p>Prevention and appraisal costs are usually lower than the cost of internal or external failure.</p>
</blockquote>
<blockquote>
<p>Acceptance criteria and UAT help ensure that deliverables meet user and stakeholder expectations.</p>
</blockquote>
<h3 id="communication">5.11 Communication</h3>
<blockquote>
<p>Communication should be planned based on stakeholder information needs, not based on habit.</p>
</blockquote>
<blockquote>
<p>The project manager should decide who needs what information, why they need it, how often they need it, and through which channel.</p>
</blockquote>
<blockquote>
<p>Interactive communication is best for discussion and decision-making; push communication is useful for updates; pull communication is useful for shared documents and repositories.</p>
</blockquote>
<blockquote>
<p>Poor communication can turn a manageable issue into a stakeholder conflict.</p>
</blockquote>
<h3 id="risk">5.12 Risk</h3>
<blockquote>
<p>Risk is an uncertain event or condition that can affect project objectives.</p>
</blockquote>
<blockquote>
<p>Risk management should be proactive, because early mitigation is usually more effective than late correction.</p>
</blockquote>
<blockquote>
<p>A risk response should match the probability, impact, urgency, and risk appetite of the organization.</p>
</blockquote>
<blockquote>
<p>Every significant risk should have an owner, a response plan, and a trigger for review.</p>
</blockquote>
<h3 id="issue--roam">5.13 Issue / ROAM</h3>
<blockquote>
<p>A risk is something that may happen; an issue is something that has already happened.</p>
</blockquote>
<blockquote>
<p>ROAM helps the team organize issues by whether they are resolved, owned, accepted, or mitigated.</p>
</blockquote>
<blockquote>
<p>Assigning an issue owner is necessary because problems do not resolve themselves through reporting alone.</p>
</blockquote>
<h3 id="execution">5.14 Execution</h3>
<blockquote>
<p>Execution is where the project manager coordinates people, resources, communication, procurement, and quality activities to produce the deliverables.</p>
</blockquote>
<blockquote>
<p>Tracking methods such as Gantt charts, roadmaps, burndown charts, and status reports help maintain transparency.</p>
</blockquote>
<blockquote>
<p>A status report should connect completed work, upcoming work, risks, issues, owners, and overall project status.</p>
</blockquote>
<h3 id="monitoring-and-controlling">5.15 Monitoring and Controlling</h3>
<blockquote>
<p>Monitoring and controlling compares actual performance against the approved baselines.</p>
</blockquote>
<blockquote>
<p>The purpose is not only to report status, but to identify variance and take timely corrective action.</p>
</blockquote>
<blockquote>
<p>If the project deviates from scope, schedule, or cost baselines, the project manager should assess the impact and decide whether a change request is needed.</p>
</blockquote>
<blockquote>
<p>Control scope helps prevent scope creep and gold plating.</p>
</blockquote>
<h3 id="evm--spi--cpi">5.16 EVM / SPI / CPI</h3>
<blockquote>
<p>Earned value management integrates scope, schedule, and cost performance.</p>
</blockquote>
<blockquote>
<p>CPI below 1 indicates that the project is spending more than the value of work earned.</p>
</blockquote>
<blockquote>
<p>SPI below 1 indicates that the project is progressing slower than planned.</p>
</blockquote>
<blockquote>
<p>EVM numbers should lead to management action, not just calculation.</p>
</blockquote>
<h3 id="procurement--sow">5.17 Procurement / SOW</h3>
<blockquote>
<p>A Statement of Work defines what the vendor must deliver and what is outside the contract scope.</p>
</blockquote>
<blockquote>
<p>Clear out-of-scope items are important because they reduce ambiguity, disputes, and scope creep.</p>
</blockquote>
<blockquote>
<p>Procurement should be controlled by monitoring seller performance against the agreement.</p>
</blockquote>
<h3 id="change-control">5.18 Change Control</h3>
<blockquote>
<p>Not every requested change should be accepted immediately, even if it seems useful.</p>
</blockquote>
<blockquote>
<p>A change request should be evaluated against scope, schedule, cost, quality, risk, and stakeholder value.</p>
</blockquote>
<blockquote>
<p>Approved changes should update the relevant baselines and be communicated to stakeholders.</p>
</blockquote>
<h3 id="closing">5.19 Closing</h3>
<blockquote>
<p>Closing is not simply stopping work; it is the formal confirmation that deliverables have been accepted and project obligations have been completed.</p>
</blockquote>
<blockquote>
<p>A closeout report should document performance, lessons learned, open items, next steps, and archived records.</p>
</blockquote>
<blockquote>
<p>Lessons learned turn project experience into organizational knowledge.</p>
</blockquote>
<blockquote>
<p>Handover is important because the deliverable may need to be operated, maintained, or supported by another team after the project ends.</p>
</blockquote>
<h3 id="tailoring--spotify-model">5.20 Tailoring / Spotify Model</h3>
<blockquote>
<p>Project management practices should be tailored to the context rather than copied mechanically.</p>
</blockquote>
<blockquote>
<p>The lesson from Spotify is not the labels of squads and tribes, but the balance between autonomy, alignment, collaboration, and continuous improvement.</p>
</blockquote>
<blockquote>
<p>A model that works in one organization may fail in another if the culture, team size, product structure, or communication needs are different.</p>
</blockquote>
<h3 id="multi-criteria-analysis">5.21 Multi-Criteria Analysis</h3>
<blockquote>
<p>Multi-Criteria Analysis supports decisions when several criteria conflict with one another.</p>
</blockquote>
<blockquote>
<p>The process makes decision assumptions visible through criteria, scoring, weighting, and ranking.</p>
</blockquote>
<blockquote>
<p>MCA improves transparency, but it does not remove judgment because scoring and weighting still require assumptions.</p>
</blockquote>

<h2 id="题型模板">6. 题型模板</h2>
<h3 id="“discuss-the-importance-of...”-类型">6.1 “Discuss the importance of…” 类型</h3>
<p>答题结构：</p>
<ol>
<li>Define the role of the process in project success。</li>
<li>Explain what happens if it is missing。</li>
<li>Link to stakeholders, baselines, risk, and value。</li>
<li>Give one applied example。</li>
<li>Conclude with PM action。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>[Process] is important because it helps the project manager [main function]. Without it, the project may suffer from [risk/problem]. In practice, it supports [stakeholder alignment / baseline control / risk response / value delivery]. Therefore, the project manager should use [tool/process] to [action] and monitor [metric].</p>
</blockquote>
<p>适用例子：</p>
<ul>
<li>Why initiation is important。</li>
<li>Why communication planning is important。</li>
<li>Why risk management is important。</li>
<li>Why project closing is important。</li>
</ul>
<h3 id="“how-would-you-manage...”-类型">6.2 “How would you manage…” 类型</h3>
<p>答题结构：</p>
<ol>
<li>Clarify objectives and value。</li>
<li>Identify stakeholders。</li>
<li>Define scope and deliverables。</li>
<li>Build plan and baselines。</li>
<li>Manage risks and communication。</li>
<li>Monitor, control, close。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>I would manage the project in stages. First, I would clarify the objectives and success criteria. Second, I would identify stakeholders and decision rights. Third, I would define the scope and break it into a WBS. Fourth, I would develop schedule and cost baselines. During execution, I would use communication, risk management, issue logs, and performance reporting to keep the project aligned. Finally, I would close the project through formal acceptance, handover, and lessons learned.</p>
</blockquote>
<h3 id="“evaluate--recommend...”-类型">6.3 “Evaluate / Recommend…” 类型</h3>
<p>答题结构：</p>
<ol>
<li>State evaluation criteria。</li>
<li>Compare options。</li>
<li>Discuss trade-offs。</li>
<li>Recommend with justification。</li>
<li>State risks and monitoring。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>I would evaluate the options using criteria such as value, feasibility, cost, schedule impact, quality, risk, and stakeholder acceptance. The best option is not necessarily the cheapest or fastest one, but the one that best supports project objectives under the given constraints.</p>
</blockquote>
<p>适用：</p>
<ul>
<li>选择 vendor。</li>
<li>选择 development approach。</li>
<li>选择 risk response。</li>
<li>选择 technical solution。</li>
</ul>
<h3 id="“compare-and-contrast...”-类型">6.4 “Compare and contrast…” 类型</h3>
<p>答题结构：</p>
<ol>
<li>State common purpose。</li>
<li>Explain difference。</li>
<li>Explain when each is useful。</li>
<li>Give project example。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>Both [A] and [B] support project management decisions, but they serve different purposes. [A] is mainly used for [purpose], while [B] is mainly used for [purpose]. In a project, the manager should use them together when [condition], because [reason].</p>
</blockquote>
<h3 id="“the-project-is-delayed--over-budget--poor-quality”-类型">6.5 “The project is delayed / over budget / poor quality” 类型</h3>
<p>答题结构：</p>
<ol>
<li>Do not blame first; diagnose。</li>
<li>Compare actual data with baseline。</li>
<li>Identify root cause。</li>
<li>Decide action: corrective, preventive, change request。</li>
<li>Communicate and monitor。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>I would first compare actual performance against the approved baselines to determine whether the problem is caused by scope change, unrealistic estimates, resource constraints, quality defects, or external factors. Then I would assign owners, update the issue log, communicate the impact to stakeholders, and decide whether corrective action or formal change control is required.</p>
</blockquote>
<h3 id="“stakeholder-conflict”-类型">6.6 “Stakeholder conflict” 类型</h3>
<p>答题结构：</p>
<ol>
<li>Identify stakeholders and interests。</li>
<li>Analyze power and impact。</li>
<li>Clarify decision rights。</li>
<li>Use communication and negotiation。</li>
<li>Escalate only if needed。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>I would treat the conflict as a stakeholder alignment issue. The project manager should first understand each stakeholder’s interest, power, expectations, and concerns. Then the team should clarify decision rights, communicate trade-offs, and seek a solution that protects project value while respecting constraints.</p>
</blockquote>
<h3 id="“scope-change”-类型">6.7 “Scope change” 类型</h3>
<p>答题结构：</p>
<ol>
<li>Identify whether it is scope creep or valid change。</li>
<li>Assess value and impact。</li>
<li>Use change request。</li>
<li>Update baseline if approved。</li>
<li>Communicate decision。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>I would not add the new work informally. I would evaluate whether it supports the project value and assess its impact on scope, schedule, cost, quality, risk, and stakeholders. If justified, it should go through formal change control and the relevant baselines should be updated.</p>
</blockquote>
<h3 id="“risk-response”-类型">6.8 “Risk response” 类型</h3>
<p>答题结构：</p>
<ol>
<li>Identify risk event。</li>
<li>Probability and impact。</li>
<li>Risk owner。</li>
<li>Response strategy。</li>
<li>Trigger and monitoring。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>I would assess each risk using probability and impact. High-probability and high-impact risks require active mitigation or avoidance, while low-impact risks may be accepted with monitoring. Each major risk should have an owner and a defined response.</p>
</blockquote>
<h3 id="“closing--lessons-learned”-类型">6.9 “Closing / lessons learned” 类型</h3>
<p>答题结构：</p>
<ol>
<li>Confirm acceptance。</li>
<li>Close contracts/accounts。</li>
<li>Handover。</li>
<li>Final report。</li>
<li>Lessons learned and archive。</li>
</ol>
<p>万能模板：</p>
<blockquote>
<p>The project should be closed only after deliverables are formally accepted, contractual obligations are completed, records are archived, and operational ownership is transferred. The closeout report should also capture lessons learned so future projects can improve.</p>
</blockquote>

<h2 id="常见对比题速记">7. 常见对比题速记</h2>
<h3 id="smart-vs-okr">SMART vs OKR</h3>
<p>万能句：</p>
<blockquote>
<p>SMART is useful for making a goal clear and measurable, while OKR links an ambitious objective with measurable key results.</p>
</blockquote>
<p>区别：</p>
<ul>
<li>SMART: 检查一个 goal 是否具体、可衡量、可实现、相关、有时限。</li>
<li>OKR: 用 Objective 指方向，用 Key Results 衡量结果。</li>
<li>SMART 更像目标质量检查。</li>
<li>OKR 更像目标管理系统。</li>
</ul>
<h3 id="raci-vs-organization-chart">RACI vs Organization Chart</h3>
<p>万能句：</p>
<blockquote>
<p>An organization chart shows reporting lines, while RACI shows task-level responsibilities and decision rights.</p>
</blockquote>
<p>区别：</p>
<ul>
<li>Org chart: 谁向谁汇报。</li>
<li>RACI: 谁做事、谁负责最终结果、咨询谁、通知谁。</li>
</ul>
<h3 id="wbs-vs-gantt-chart">WBS vs Gantt Chart</h3>
<p>万能句：</p>
<blockquote>
<p>WBS defines what work must be done; a Gantt chart defines when the work will be done.</p>
</blockquote>
<p>区别：</p>
<ul>
<li>WBS: scope decomposition。</li>
<li>Gantt: schedule visualization。</li>
<li>WBS 先于 Gantt。</li>
</ul>
<h3 id="risk-vs-issue">Risk vs Issue</h3>
<p>万能句：</p>
<blockquote>
<p>A risk may happen; an issue has already happened.</p>
</blockquote>
<p>区别：</p>
<ul>
<li>Risk: 未来不确定事件。</li>
<li>Issue: 已经发生的问题。</li>
<li>Risk register 管风险；issue log 管问题。</li>
</ul>
<h3 id="cpi-vs-spi">CPI vs SPI</h3>
<p>万能句：</p>
<blockquote>
<p>CPI shows cost efficiency, while SPI shows schedule efficiency.</p>
</blockquote>
<p>区别：</p>
<ul>
<li>CPI &lt; 1: over budget。</li>
<li>SPI &lt; 1: behind schedule。</li>
<li>两者都 &lt; 1：又超支又延误。</li>
</ul>
<h3 id="quality-vs-grade">Quality vs Grade</h3>
<p>万能句：</p>
<blockquote>
<p>Quality means meeting requirements and being fit for use; grade refers to the category or feature level of the product.</p>
</blockquote>
<p>区别：</p>
<ul>
<li>Low grade can still be high quality。</li>
<li>High grade can still be low quality if it fails requirements。</li>
</ul>
<h3 id="accuracy-vs-precision">Accuracy vs Precision</h3>
<p>万能句：</p>
<blockquote>
<p>Accuracy refers to correctness, while precision refers to exactness.</p>
</blockquote>
<p>区别：</p>
<ul>
<li>Accurate: 接近真实值。</li>
<li>Precise: 表达很精确，但不一定正确。</li>
</ul>
<h3 id="scope-creep-vs-gold-plating">Scope Creep vs Gold Plating</h3>
<p>万能句：</p>
<blockquote>
<p>Scope creep means uncontrolled expansion of scope; gold plating means doing more than required even without approval.</p>
</blockquote>
<p>区别：</p>
<ul>
<li>Scope creep: 外部/内部不断加需求。</li>
<li>Gold plating: 团队主动做超出要求的东西。</li>
<li>都需要 scope control。</li>
</ul>
<h3 id="predictive-vs-adaptive-vs-hybrid">Predictive vs Adaptive vs Hybrid</h3>
<p>万能句：</p>
<blockquote>
<p>Predictive is suitable when requirements are stable; adaptive is suitable when requirements are uncertain; hybrid combines both when different parts of the project have different levels of uncertainty.</p>
</blockquote>
<p>适用：</p>
<ul>
<li>Predictive: construction, regulation, fixed contract, stable scope。</li>
<li>Adaptive: software, innovation, changing requirements。</li>
<li>Hybrid: infrastructure + smart technology + stakeholder feedback。</li>
</ul>

<h2 id="考场可直接套用的短答案模板">8. 考场可直接套用的短答案模板</h2>
<h3 id="句话模板">5 句话模板</h3>
<p>适合短论述题。</p>
<ol>
<li>The main issue is [problem] because it affects [value/scope/schedule/cost/quality/stakeholder].</li>
<li>I would first clarify [objectives/stakeholders/scope/baseline].</li>
<li>The appropriate tool/process is [tool] because it helps [reason].</li>
<li>The project manager should [action], assign [owner], and communicate [information] to [stakeholder].</li>
<li>The result should be monitored by [metric] and adjusted through [change control/risk review/status report].</li>
</ol>
<h3 id="点-bullet-模板">8 点 bullet 模板</h3>
<p>适合中等论述题。</p>
<ol>
<li>Define project value。</li>
<li>Define success criteria。</li>
<li>Identify stakeholders and decision rights。</li>
<li>Define scope and deliverables。</li>
<li>Build WBS/Gantt/cost baseline。</li>
<li>Identify risks and responses。</li>
<li>Plan communication and reporting。</li>
<li>Monitor performance and close formally。</li>
</ol>
<h3 id="分钟长答模板">15 分钟长答模板</h3>
<p>适合大题。</p>
<ol>
<li>Opening: frame the project as value delivery。</li>
<li>Objectives: SMART/OKR/success criteria。</li>
<li>Stakeholders: power-interest + RACI。</li>
<li>Planning: charter, scope, WBS, schedule, cost。</li>
<li>Execution: team, quality, procurement, communication。</li>
<li>Monitoring: baseline, RAG, EVM, issue/change log。</li>
<li>Risks: probability-impact, response, owner。</li>
<li>Closing: acceptance, handover, lessons learned。</li>
<li>Conclusion: trade-off and PM role。</li>
</ol>

<h2 id="如果忘记具体概念，怎么救答案">9. 如果忘记具体概念，怎么救答案</h2>
<p>闭卷时忘记术语，不要空着。用五个通用维度救：</p>
<h3 id="value">Value</h3>
<p>写项目为什么要做，创造什么价值。</p>
<blockquote>
<p>The project should remain aligned with its intended value.</p>
</blockquote>
<h3 id="stakeholder">Stakeholder</h3>
<p>写谁影响项目，谁需要被沟通。</p>
<blockquote>
<p>Stakeholders should be analyzed by power, interest, influence, and impact.</p>
</blockquote>
<h3 id="baseline">Baseline</h3>
<p>写计划和实际要比较。</p>
<blockquote>
<p>Actual performance should be compared with approved baselines.</p>
</blockquote>
<h3 id="risk-1">Risk</h3>
<p>写不确定性和应对。</p>
<blockquote>
<p>The project manager should identify uncertainties early and prepare responses.</p>
</blockquote>
<h3 id="action">Action</h3>
<p>写责任人、时间、指标、沟通。</p>
<blockquote>
<p>Each issue should have an owner, action plan, deadline, and monitoring metric.</p>
</blockquote>
<p>只要把这五个维度写出来，答案通常会有项目管理逻辑。</p>

<h2 id="闭卷背诵小抄">10. 闭卷背诵小抄</h2>
<p>最值得背的 20 句：</p>
<ol>
<li>Project success should be evaluated by both outputs and outcomes.</li>
<li>A project can be delivered on time and still fail if it does not meet stakeholder needs.</li>
<li>The project manager should clarify objectives before detailed planning.</li>
<li>SMART goals reduce ambiguity by defining target, metric, feasibility, relevance, and deadline.</li>
<li>Objectives define direction; key results define measurable success.</li>
<li>Stakeholder engagement should be prioritized by power, interest, influence, and impact.</li>
<li>RACI separates who does the work from who owns the final decision.</li>
<li>WBS defines what work must be done; Gantt defines when it will be done.</li>
<li>Planning creates baselines for later monitoring and control.</li>
<li>Communication should be tailored to stakeholder information needs.</li>
<li>Risk responses should match probability, impact, urgency, and risk appetite.</li>
<li>A risk may happen; an issue has already happened.</li>
<li>Scope change should be evaluated before being approved.</li>
<li>Control scope prevents scope creep and gold plating.</li>
<li>Quality should be built into the process, not inspected only at the end.</li>
<li>CPI below 1 means over budget; SPI below 1 means behind schedule.</li>
<li>EVM numbers should lead to management action, not just calculation.</li>
<li>A SOW should define scope, out-of-scope, deliverables, milestones, and payment terms.</li>
<li>Closing requires formal acceptance, handover, archive, and lessons learned.</li>
<li>Project management practices should be tailored to context, not copied mechanically.</li>
</ol>

<h2 id="最推荐的答题风格">11. 最推荐的答题风格</h2>
<p>不要这样写：</p>
<blockquote>
<p>Risk management is the process of identifying and managing risks. It is important.</p>
</blockquote>
<p>要这样写：</p>
<blockquote>
<p>In this project, risk management is necessary because schedule delay, cost overrun, and stakeholder dissatisfaction can directly affect the project value. I would identify specific risk events, assess probability and impact, assign risk owners, and choose responses such as mitigation, transfer, acceptance, or escalation. The risk register should be reviewed regularly and linked to the communication plan and status report.</p>
</blockquote>
<p>不要这样写：</p>
<blockquote>
<p>Communication is important because people need information.</p>
</blockquote>
<p>要这样写：</p>
<blockquote>
<p>Communication should be planned according to stakeholder information needs. Senior leaders need concise status and escalation points, while the delivery team needs frequent coordination. Therefore, I would combine interactive meetings for decision-making, push communication for updates, and pull communication for shared documents.</p>
</blockquote>
<p>不要这样写：</p>
<blockquote>
<p>WBS is breaking work into smaller parts.</p>
</blockquote>
<p>要这样写：</p>
<blockquote>
<p>WBS is a scope control tool. It decomposes the approved project scope into manageable work packages, which later become activities in the schedule and cost items in the budget. This prevents missing work and helps control scope creep.</p>
</blockquote>

<h2 id="最终记忆模型">12. 最终记忆模型</h2>
<p>把整门课压缩成一条线：</p>
<blockquote>
<p>Initiation defines why the project exists. Planning defines what will be done and how it will be measured. Execution produces the deliverables. Monitoring and controlling compares actual performance against baselines and manages change. Closing confirms acceptance, transfers ownership, and captures lessons learned.</p>
</blockquote>
<p>再压缩成 6 个关键词：</p>
<blockquote>
<p>Value, Stakeholders, Scope, Baseline, Risk, Action.</p>
</blockquote>
